## 2. 项目背景与目标

### 2.1 背景

运营团队在 **Temu 平台** 上进行商品上新时，需要填写一个属性字段——**货号**（实际对应内部商家编码）。当 Temu 平台向仓库下发备货单时，通过该货号与仓库 ERP 系统中的商品数据关联，从而让仓库人员明确备货单应发哪些商品。

由于运营每天**大量上新**，导致每天需要**创建大量货号商品**。ERP 管理员负责的团队众多，**手工逐条创建工作量极大**，效率低下且容易出错。因此，亟需一个**批量建货号工具**来提升效率。

### 2.2 目标

- **核心目标**：提供一个支持批量导入、自动调用快麦 ERP 接口完成货号创建的工具，将人工逐条操作转为自动化批量处理。
- **效率目标**：将每日大量货号创建的时间成本从数小时级降低至分钟级。
- **质量目标**：减少人工录入错误，确保货号与 ERP 系统准确关联，并记录每条数据的处理状态和失败原因。

## 4. 术语与规则定义

### 4.1 货号编码规则

货号格式示例：`69-39-BSG07`

| 分段 | 示例 | 含义 |
|------|------|------|
| `69` | 69 | **固定开头** |
| `39` | 39 | **品牌内部编码**（如 wkau 对应 39） |
| `BSG` | BSG | **产品名称简写**（如补色膏） |
| `07` | 07 | **系统自增序号**（当前品牌下的第7个） |

### 4.2 ERP 套装商品命名规则

**套装名称格式**：

```
品牌 + 原品名称 * 数量 + 单品 * 数量 + …
```

> 例如：`wkau + 补色膏 * 1 + 自粘袋 * 1 + …`

### 4.3 业务对象定义

| 对象 | 定义 | 说明 |
|------|------|------|
| **贴纸** | 一种固定分类的单品 SKU | 用于套装组合，编码格式为商家编码，名称格式：`品牌+产品名称+容量+"贴纸"` |
| **套装** | 不含 SKU 的套件 | 由多个单品（含贴纸）组合而成，作为最终对外销售的货号商品 |
| **单品** | 系统已存在的基础 SKU | 在创建套装时直接选取，无需新建 |

### 5.3 详细功能描述

#### 5.3.1 Excel 导入与数据解析

**输入**：从飞书多维表格导出的 Excel 文件。

**表格预期字段**（需确认）：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 品牌 | 文本 | 是 | 如 wkau |
| 产品名称 | 文本 | 是 | 如补色膏 |
| 容量 | 文本 | 否 | 用于贴纸名称 |
| 成分 | 文本 | 否 | 套装属性 |
| 执行标准 | 文本 | 否 | 套装属性 |
| 配件清单 | 文本 | 是 | 套装所需的单品 |
| 图片 | 文本/链接 | 否 | 贴纸图片 URL 或路径 |
| 品牌编码 | 文本 | 是 | 如 39（对应 wkau） |
| 产品简写 | 文本 | 是 | 如 BSG |

**处理逻辑**：
1. 用户上传 Excel 文件。
2. 系统解析并展示预览（前 N 条数据）。
3. 支持字段映射调整（如用户表格列名与系统预期不一致时手动映射）。

#### 5.3.2 货号编码生成规则

**规则**：
- 固定前缀：`69`
- 品牌编码：从品牌配置表读取（如 wkau → 39）
- 产品简写：从表格或配置表读取（如 BSG）
- 自增序号：查询快麦 ERP 当前品牌下已有货号数量，自动递增

**编码公式**：
```
货号 = 69 + "-" + 品牌编码 + "-" + 产品简写 + 自增序号(2位)
```

> 示例：`69-39-BSG07`

#### 5.3.3 两步创建流程

创建货号商品实际需要两步操作：

**第一步：新增贴纸**

| 属性 | 值 |
|------|------|
| 分类 | 固定为 "贴纸" |
| 商家编码 | 贴纸编码（由系统生成或表格指定） |
| 名称 | `品牌 + 产品名称 + 容量 + "贴纸"` |
| 图片 | 有则上传，无则忽略 |

**第二步：新增套装**

| 属性 | 值 |
|------|------|
| 类型 | 不含 SKU 套件 |
| 分类 | 固定（需确认具体分类值） |
| 主商家编码 | 系统生成的不重复编码（即最终货号） |
| 商品名称 | `品牌 + 产品名称 * 1 + 自粘袋 * 1 + …`（按配件表拼接） |
| 成分 | 表格提供 |
| 执行标准 | 表格提供 |
| 套件单品 | 表格需要的配件 + 第一步新增的贴纸单品 |

#### 5.3.4 快麦 ERP 接口调用

**接口来源**：快麦开放平台
- 文档地址：`https://open.kuaimai.com/docs/api/API文档/...`
- 主要接口：**修改/新增普通商品**（POST）

**认证方式**（二选一）：
1. 从 Cookie 中获取令牌进行调用（模拟人工登录态）。
2. 通过快麦开放平台正式 API 授权调用（推荐，更稳定）。

**调用策略**：
- 如果采用**预置货号方案**（商品编码由系统预先生成，不依赖 ERP 自增序号），则仅需调用**新增商品接口**即可。
- 如需获取 ERP 自增序号，则额外需要调用**查询接口**。

#### 5.3.5 结果回传与状态记录

**成功流程**：
1. 调用 ERP 接口成功创建商品后，将生成的**货号（商家编码）**回填到 Excel 的「商品货号」列。
2. 记录该条数据状态为「创建成功」。

**失败流程**：
1. 调用失败时，记录状态为「创建失败」。
2. 记录失败原因（如：接口返回错误、参数缺失、编码重复等）。
3. 支持失败重试机制。

**后续操作**：
- 用户可从处理结果中批量复制商家编码，回填到飞书多维表格，完成业务流程闭环。

## 8. 非功能需求

| 类型 | 需求 | 说明 |
|------|------|------|
| **性能** | 支持一次性处理 ≥200 条数据 | 满足日常批量上新的峰值需求 |
| **稳定性** | 接口调用失败支持重试（最多 3 次） | 避免因网络波动导致批量失败 |
| **安全性** | 令牌/ Cookie 信息不落地存储 | 调用完成后即时清理敏感凭证 |
| **易用性** | 支持拖拽上传 Excel | 降低操作门槛 |
| **可扩展性** | 品牌配置表与配件表支持动态维护 | 后续新增品牌/配件无需发版 |
| **日志** | 记录每条数据的处理日志 | 便于问题排查和审计 |

## 9. 技术方案建议

### 9.1 实现形式

| 方案 | 优点 | 缺点 | 建议 |
|------|------|------|------|
| **网页端** | 跨平台、无需安装、易于分享 | 需要部署服务器 | **推荐** |
| **客户端** | 可本地操作、安全性高 | 需要维护多平台版本 | 备选 |
| **脚本工具** | 开发快、灵活 | 使用门槛高 | 可内部先用 |

### 9.2 接口调用方案

| 方案 | 说明 | 建议 |
|------|------|------|
| **方案 A：Cookie 模拟** | 从浏览器 Cookie 获取令牌，直接调用 ERP 内部 HTTP 接口 | 快速实现，但稳定性受限于登录态 |
| **方案 B：开放平台 API** | 通过 `open.kuaimai.com` 正式接口调用 | 更规范、稳定，推荐长期方案 |
